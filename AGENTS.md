# AGENTS.md — CutList Optimizer

Coding-agent reference for this repository. Read before making changes.

---

## Stack

- **Runtime**: React 19 + TanStack Start (SSR framework built on Vite)
- **Routing**: TanStack Router — file-based routes under `src/routes/`
- **State**: `useReducer` in `src/hooks/useCutOptimizer.ts`; no global store
- **Styling**: Tailwind CSS v4 (no config file — uses `@tailwindcss/vite` plugin)
- **Testing**: Vitest v3
- **Language**: TypeScript 5.7, strict mode, `verbatimModuleSyntax`

## TypeScript Rules

The `tsconfig.json` enforces:

- `strict: true` — all strict checks on
- `noUnusedLocals: true` — unused variables are **errors**, not warnings
- `noUnusedParameters: true` — unused function params are errors  
  (prefix with `_` to suppress: `_unusedParam`)
- `verbatimModuleSyntax: true` — use `import type` for type-only imports
- `noFallthroughCasesInSwitch: true`

Always run `npx tsc --noEmit` after edits and fix all errors before finishing.

## Path Aliases

```typescript
import type { SheetDef } from '#/lib/cutOptimizer'  // ✓ preferred
import type { SheetDef } from '../lib/cutOptimizer'  // ✗ avoid relative from components
```

`#/*` resolves to `src/*`. Use it for all cross-directory imports.  
Relative imports (`./foo`) are fine within the same directory.

## Code Style

### General

- **2-space indentation**, no tabs
- **Single quotes** for strings in TypeScript/TSX
- **No semicolons** — the codebase is semicolon-free
- Line length: aim for ≤ 100 chars; section-header comments may be longer
- Trailing commas in multi-line arrays/objects/params

### Naming

| Construct | Convention | Example |
|-----------|-----------|---------|
| Types / Interfaces | `PascalCase` | `SheetDef`, `ColPlan` |
| Functions | `camelCase` | `buildColPlans`, `packCol` |
| React components | `PascalCase` default export | `ResultsPanel` |
| Constants (module-level) | `UPPER_SNAKE` or `PascalCase` | `ALGORITHMS`, `PART_COLORS` |
| Boolean vars | `is`/`has`/`can` prefix | `isCalculating`, `heightFits` |
| Unused params | `_` prefix | `_unusedParam` |

### Section Comments

Long files use banner comments to divide sections:

```typescript
// ─── Section Name ─────────────────────────────────────────────────────────────
```

Maintain this pattern when adding new logical sections to `cutOptimizer.ts`.

### Interfaces vs Types

- Use `interface` for object shapes that may be extended
- Use `type` for unions, intersections, and aliases: `type AlgorithmId = 'guillotine' | …`

### Imports Order (manual, no auto-sort)

1. React / framework imports
2. Third-party libraries
3. Internal `#/` path-alias imports
4. Relative `./` imports
5. `import type` statements last (or co-located with their value import)

## React Patterns

- **State management**: `useReducer` + dispatched actions (see `useCutOptimizer.ts`)
  - All optimizer state mutations go through the reducer — no direct `useState` for
    shared state
  - `useCallback` wraps every dispatch helper to stabilise references
- **Components**: functional only, no class components
- **Props**: always type with an explicit `interface FooProps { … }` above the component
- **Effects**: use `useEffect` + `ResizeObserver` pattern (see `ResultsPanel.tsx`) for
  DOM measurement; clean up in the return function

## Core Library: `cutOptimizer.ts`

This file is the heart of the project. Key rules:

- **Do not change** the public API of `optimizeCutList()` — only extend its return type
- **Do not alter** `GuillotinePacker`, `MaxRectsPacker`, or `ShelfPacker` placement logic
- Internal helpers (`buildColPlans`, `packCol`, `planTrackSawSheet`, etc.) are fair game
- `ExpandedPart` interface must be declared **before** the Track Saw section (it is used
  by helpers defined before `optimizeCutList`)
- All dimensions are in **millimetres** (integers in practice, but typed as `number`)
- `kerf` is additive: a part of width `w` occupies `w + kerf` mm of sheet space

### Adding a new algorithm

1. Add the ID to `AlgorithmId` union type
2. Add an entry to the `ALGORITHMS` record
3. Add a packer class (or inline logic) above the `// ─── Validation` section
4. Add a branch in `optimizeCutList` (see the `tracksaw` fast-path as a template)
5. Add tests in `src/lib/__tests__/cutOptimizer.test.ts`

## Testing

- Framework: **Vitest** (imported as `import { describe, it, expect } from 'vitest'`)
- All tests live in `src/lib/__tests__/`
- Test helpers (e.g. `noOverlap`) are defined at the top of the test file — reuse them
- `import type` must be used for type-only test imports (required by `verbatimModuleSyntax`)
- The `noOverlap(placements, kerf)` helper is the canonical overlap check — always use it
  when asserting placement correctness
- Do **not** use `npx jest` — only `npm test` (Vitest, not Jest)

## Constraints / Do-Nots

- **No new npm dependencies** — the package set is intentional; avoid adding packages
- **Do not edit `src/routeTree.gen.ts`** — it is auto-generated
- **Do not edit files in `src/integrations/`** unless wiring a new integration
- **Do not use `cd dir && command`** — if running shell commands, use the `workdir`
  parameter or equivalent
- The `cuts: CutLine[]` field on `SheetResult` is populated by `deriveCutSequence()` —
  every algorithm's sheet results must go through this call in `optimizeCutList`
